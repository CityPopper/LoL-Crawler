# Phase 05 — Resilience Layer

**Role:** Product Manager
**Objective:** The system handles all documented failure modes without data loss. Recovery and Delay Scheduler services are complete, tested, and running in Docker. The pipeline is production-safe.

**Value unlocked:** API key rotation incidents, transient 5xx errors, and rate limit spikes no longer require manual intervention. The system self-heals.

**Complexity: MEDIUM-HIGH** — Recovery implements multiple failure routing paths; Delay Scheduler has timing-sensitive logic. The interaction between Recovery → delayed:messages → Delay Scheduler → source stream is the most complex end-to-end path in the system. Requires careful review of edge cases.

- AC-05-04: `failure_code="http_5xx"`, incoming `dlq_attempts=2` (same boundary): archived to `stream:dlq:archive`; ACK from `stream:dlq`. If `payload.match_id` present: `HGET match:{match_id} status` = `"failed"`; `SISMEMBER match:status:failed {match_id}` = 1.

- AC-05-06: `failure_code="http_404"`: logged at WARNING; ACK from `stream:dlq`; `ZCARD delayed:messages` unchanged; `XLEN stream:dlq:archive` unchanged (discard, no archive).

## Notes

- Delay Scheduler does not check `system:halted`. This is an explicit design decision: messages in `delayed:messages` continue moving to their target streams even when the system is halted. Services won't process them because they exit their loops when halted. This prevents messages from being stranded in `delayed:messages` indefinitely.
- `delayed:messages` ZSET member uniqueness: each envelope stored as member includes `enqueued_at` timestamp. If the same message is nack'd twice before the first entry is moved (e.g., crash between ZADD and ACK), two entries may exist for the same logical message. Both will be moved to the source stream and processed. Downstream idempotency handles duplicates. This is an accepted limitation — the alternative (using a canonical key as member and storing the envelope separately) adds complexity not justified at this scale.
- Recovery continues processing `stream:dlq` even when `system:halted=1`. This is intentional — Recovery processes the 403 DLQ entries that caused the halt. Only stream-consuming services that perform Riot API calls stop.
